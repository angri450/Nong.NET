OCR Runtime 版本解耦说明

这次改动的目的很明确：以后 CLI、Word、PDF、Excel、PPT 发小版本时，不再被迫重新发布 100MB 级别的 OCR native runtime 大包。

这轮发布 / 推送 / NuGet 动作

现在是什么情况

代码已经改完并验证通过，但还没有提交仓库，也没有推送 GitHub / Gitee / GitCode，更没有发布 NuGet。 工作区当前处于“实现完成、等待提交”的状态。

核心变化是新增 Cli/Common/OcrRuntimeVersion.cs，让 nong ocr install-model 使用这个独立常量查找 Angri450.Nong.OcrRuntime.* 包，而不是继续绑定 CliVersion.Current。 同时把 OCR runtime 打包工程拆到独立 Angri450.Nong.OcrRuntime 仓库，主仓库只消费已发布 runtime 包。

之前的问题

旧逻辑

OCR runtime 包版本直接使用 CliVersion.Current。

结果是 CLI 从 4.0.0 升到 4.0.1 时， install-model 会去找 Angri450.Nong.OcrRuntime.WinX64 4.0.1。

→

实际后果

哪怕只是修了 Word 或 PDF 的一个小 bug，也要重新打包、上传、等待镜像同步 OCR native bundle。

这个包很大，且 native 内容没变时重复发布没有价值。

现在的方案

CLI 版本

CliVersion.Current 继续代表 CLI 自己的版本，用在命令输出的 meta.version 等 CLI 合同里。

+

OCR Runtime 版本

OcrRuntimeVersion.Current 专门代表 native runtime bundle 版本。 当前值是 4.0.0，对应独立 Angri450.Nong.OcrRuntime 仓库已发布的 runtime 版本。

发版规则： 普通 CLI / Word / PDF / Excel / PPT 小版本发布不动 OCR runtime。 只有 Paddle/OpenCV native 文件或 runtime 安装合同变化时，才 bump OCR runtime 版本并重新推大包。

我们具体改了什么

  • 新增 Cli/Common/OcrRuntimeVersion.cs，集中锁定 OCR runtime 包版本。
  • 把 OcrCommands.cs 中 5 个 first-party runtime 包从 CliVersion.Current 改为 OcrRuntimeVersion.Current。
  • 新建独立 Angri450.Nong.OcrRuntime 仓库，维护 WinX64、LinuxX64、LinuxArm64、OsxX64、OsxArm64 五个平台包。
  • 补充测试，防止以后又把 runtime package plan 绑回 CLI 版本。
  • 更新 README、CLAUDE、release-checklist、CAPABILITY、CLI AGENT 和 runtime 仓库文档。
  • 补充 changelog，记录这次版本策略调整。

验证过什么

  • dotnet build .\Cli\NongCli.csproj -c Release 通过。
  • OCR 相关测试通过：15 passed。
  • 完整 Cli.Tests 通过：116 passed。
  • nong ocr install-model pp-ocrv5-mobile --dry-run --json 通过。
  • dry-run 输出中 runtimePackage.version 为独立 runtime 版本，meta.version 仍为 CLI 版本。

以后怎么判断要不要推 OCR Runtime

场景 是否推 OCR runtime 大包 原因
只修 CLI 命令、Word 格式、PDF 切片、Excel/PPT 合同 不推 native Paddle/OpenCV 内容没有变化，继续使用已发布 runtime bundle。
只改 README、skill、changelog、agent 指南 不推 文档变化不影响 OCR runtime 安装物。
升级 PaddleInference、OpenCvSharp native 文件 要推 实际 native bundle 内容变化，必须 bump OCR runtime 版本。
修改 runtime 解包目录、manifest 合同、安装校验规则 视情况推 如果旧 runtime 包不能满足新安装合同，就要 bump 并重发。

剩余风险
