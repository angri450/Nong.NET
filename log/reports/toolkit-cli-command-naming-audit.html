一句话结论

现在可以继续用这套命令面。 96 个命令已经能覆盖 Toolkit 的主要路由，命令分组也基本符合“Word 处理文档、Inspect 处理论文/公文语义、Chart/Excel 处理实验数据、Slice 读统一切片、Skill 管技能生命周期”的边界。

需要处理的是命令的“可理解性债务”：有些名字短、省 token，但新人看不出完整含义，例如 inspect refs、inspect varplan、inspect data-req；有些名字容易被误认为能力更大，例如 word preview、word rebuild、chart analyze。这些不需要马上破坏性改名，优先补文档和 alias。

三分法总判定

这次问题不是单一类型。 当前命令面里，“重复”很少，“缺少”是真正的后续功能路线图，“不对”主要是名字或描述不够准。处理顺序应该是：先治理名字不准，再规划缺少功能，最后只对真重叠命令做兼容收口。

类别结论代表项处理方式
命令重复只有 1 组是真功能重叠；11 个 word add-* 是兼容 alias，不算坏重复。word extract 与 word images --output 有重叠；word add paragraph 与 word add-paragraph 是兼容入口。定 canonical 命令，保留 alias；不要马上删命令。
命令缺少缺的是新能力，不是名字问题。pptx create/edit、Excel 样式/公式/透视、Chart 箱线图/热力图/直方图、PDF 合并/拆分/OCR PDF、inspect write-letter、Word compare/视觉预览。另开功能 plan，审底层库，再补 CLI、Manifest、测试、Toolkit。
命令不对没有阻断级错误，但有一批名字或描述会误导。word preview、word rebuild、inspect refs、inspect varplan、inspect data-req、chart analyze、ocr analyze-image、progress report。优先补更准确 alias 和描述；Toolkit 教新名字，旧名字兼容。

1. 命令重复：哪些是真的，哪些不是

命令关系判定原因建议
word extract vs word images --output真重叠两者都能处理 DOCX 内嵌图片；images 还承担“列出图片”的语义，所以覆盖面更自然。把 word images 定为 canonical；word extract 保留兼容，后续可标注 legacy。
word add paragraph vs word add-paragraph 等 11 个兼容 alias扁平命令是老脚本兼容入口，嵌套命令才是现在教学的 canonical form。不删。Toolkit 只教 word add paragraph 这类嵌套命令。
inspect diagnose vs inspect classify/structure/evidence/data-req/gap/semantics不是重复diagnose 是总诊断，其他是低 token 分步诊断。保留。文档说明“总命令 vs 分步命令”。
chart analyze vs chart anova / chart duncan不是重复analyze 是一站式处理组分析，anova/duncan 是单项统计。保留，但把 description 写窄。
word preview vs word dissect / word outline / word stats不是重复它们输出层级不同：诊断概览、完整切片、目录、计数。保留，但 preview 需要 alias 或说明。
slice inspect vs inspect diagnose不是重复一级命令不同：前者检查 NongPandoc 包契约，后者诊断论文内容。保留。

2. 命令缺少：这些是新能力，不是文档没写清

缺少领域建议命令名是否影响当前 Toolkit 主流程判断
PPTX 生成/编辑pptx create、pptx edit中高。当前 PPTX 技能只能读，不能做交付型 PPT。真缺少
Excel 高级编辑excel style、excel formula、excel pivot、excel edit中。当前能创建简单表，但不能做复杂 workbook。真缺少
科研图表类型chart boxplot、chart histogram、chart heatmap、chart radar中高。农业数据常见图种还不全。真缺少
PDF 编辑/可搜索 OCR PDFpdf merge、pdf split、pdf ocr、pdf watermark中。当前 PDF 是读、切片、渲染、提图。真缺少
Inspect 扩展写作/审核inspect write-letter、inspect official-check、lit format-refs中。论文/公文已有，信函和正式审核还没暴露。真缺少
Word 高级审阅word compare、word render-preview、word apply-style中。当前格式证据强，但不是视觉渲染或文档 diff。真缺少

3. 命令不对：不是不能用，是词没完全达意

现有命令哪里不对更准确的词处理建议
word preview会让人以为是视觉预览或页面截图，实际是结构诊断。word diagnose加 alias；description 写“structure diagnostic, not visual render”。
word rebuild“rebuild”太大，实际是清理 OOXML 样式污染。word clean-styles加 alias；Toolkit 教 clean-styles，保留 rebuild。
inspect refs容易误会成外部文献检索或引用格式生成。inspect refs-check / inspect references加 alias；明确外部检索走 lit search。
inspect varplan缩写太内部，用户不知道是变量操作化。inspect variables加 alias。
inspect data-req缩写可猜，但不适合教学。inspect data-requirements加 alias。
chart analyze范围太大，像任意统计分析。chart treatment-analyze 或更窄 description先改描述；等统计命令变多再决定 alias。
ocr analyze-image放在 ocr 组下但不做 OCR；当前靠描述兜住。image analyze 或 multimodal analyze-image短期保留；文档必须写“no OCR”。长期看是否新增一级组。
progress report可能被理解成实时进度，实际是读取 log/ 生成报告。progress log-report低优先级；当前不改也能接受。

命名原则

1. 动词必须说明副作用。 read、inspect、validate 应该只读；create、write、format、reflow 应该明确会生成新文件。

2. 名字不能承诺不存在的能力。 preview 如果不是视觉渲染，就必须在描述里写清是结构诊断；analyze 如果只做 ANOVA/Duncan/描述统计，就不要暗示任意统计分析。

3. 领域边界比短名字重要。 公文“生成草稿”属于 inspect write-official，既有 DOCX 的公文排版属于 word format-gongwen，这个分工是对的。

4. 兼容脚本靠 alias，不靠改老命令。 已暴露命令是契约，除非有明显误导和迁移方案，不要直接删改；推荐先加长名 alias，再把 Toolkit 教学改到更清楚的叫法。

优先级判断

级别含义本轮结论建议动作
P0会导致错误调用、数据丢失、技能无法工作。未发现。不做破坏性改名。
P1名字会让人误解能力边界，但现有命令可用。word preview、word rebuild、inspect refs、inspect varplan、inspect data-req、chart analyze。补 alias 或强化 description / Toolkit 文档。
P2名字还行，但领域里常见能力没覆盖。PPTX 写入、Excel 样式/公式/透视、图表箱线图/热力图/直方图、PDF 合并/拆分/OCR PDF、Inspect 信函写作等。作为功能路线图，不要用现有名字硬包。
P3只是可读性或教学顺序问题。多数命令组属于此类。保持命令稳定，调整技能说明。

建议执行顺序

顺序动作理由改动范围
1先不重命名任何现有命令。现有 96 命令已被 Toolkit 和测试引用，直接改名会制造新不对齐。无代码改动
2把 P1 命令的 description 和 Toolkit 说明补得更直白。这是最低风险修复，能马上减少误用。CLI Manifest / SKILL.md / references
3给 P1 项补 alias，并在 nong commands --json 暴露。既保留老脚本，又给新用户更好理解的入口。System.CommandLine + Manifest + contract tests
4把真实功能缺口单独立计划。PPTX 写入、Excel 编辑、Chart 新图种不是文案问题，需要底层实现和测试。按模块拆 plan
5每次 CLI 命令面变化后同步 Toolkit，再跑双向审计。防止再次出现“CLI 有但 skill 没写”或“skill 教了 CLI 没有”的问题。Nong.Cli.Net + Nong.Toolkit.Net

最终判断

当前命令面是可工作的，不需要大改。 最稳妥的路线是：命令契约保持稳定，先用 alias 和文档把语义补清楚；真正缺的能力另开功能任务。

下一轮如果要动代码，建议从 5 个 P1 alias 开始：word diagnose、word clean-styles、inspect references、inspect variables、inspect data-requirements。这批改动小、收益高、风险可控。
