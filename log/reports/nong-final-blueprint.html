先说最重要的话

Nong 不是“很多命令的 Office 工具箱”。

Nong 的本质，是复杂对象的“可解释拆解 + 可控重装”。 文档、PDF、PPT、Excel、图片，不应该作为一个黑盒整体直接丢给 AI。它们必须先被拆成 AI 能吃下去的几条信息流，再由 CLI 负责把这些信息在可接受损耗范围内装回去。

你说的“一刀三流”，不是 Word 的一个 feature，而是整个 Nong 的总产品哲学。

最终总原则

原则含义为什么重要
一刀面对复杂对象，先走一个稳定入口，把原对象切成统一中间表示。AI 不再需要自己去解 zip、读 OOXML、猜页面、手搓 XML。
三流至少分成：内容流、结构流、格式/资产流。必要时再附加诊断流。AI 可以分开看“写了什么”“怎么组织”“长什么样”，不会混成一锅。
可重装拆出来不是只为了看，还要能用来回写、修复、重排、重建。否则 Nong 只能当只读解析器，不是生产工具。
损耗可控重装回去后，内容、结构、表格、图注、图片语义、可见格式不能乱丢。用户原本是申报表，就不能被“优化”成一串没表格的文本。
统一合同Word / PDF / PPTX / Excel 的 slice 输出都走统一包形状。AI 学一套读法，就能读四类复杂对象。

三层结构，围绕一刀三流服务

层身份围绕一刀三流该做什么不该做什么
Nong.NanoBot.Net Agent 运行层 接用户意图，决定是先切片、先读包，还是先做重装；负责多轮任务编排。 不直接解析原始复杂对象，不自己拼 OOXML / PDF / PPT XML。
Nong.Toolkit.Net AI 教学层 教 AI：什么时候必须先切，读哪个流，什么证据算数，什么损耗不能接受。 不自己做底层重建，不绕过 CLI 直连库。
Nong.Cli.Net 执行层 实现所有“切、读、改、装”的硬能力。保证 JSON 合同、产物和错误码。 不负责对话，不负责模糊语义推理。

一刀三流，不是三个文件，是三类真相

1. 内容流

回答“它到底写了什么”。

  • content.nongmark：主要给人和 AI 读的增强文本投影
  • content.jsonl：逐块 canonical block 流，带稳定 ID
  • 段落、标题、表格、图注、公式、图片占位、脚注、引用都要有类型

2. 结构流

回答“这些内容是怎么组织的”。

  • structure.json：标题树、块顺序、页/节/slide/sheet 映射
  • 表格、图、标题、注释、批注、修订都要能定位
  • 块 ID 要稳定，后续插入和回写靠它

3. 格式/资产流

回答“它长什么样，有哪些素材”。

  • format.json：字体、行距、边框、样式、页面、节、版式证据
  • assets/manifest.json：图片、媒体、嵌入对象、来源、关联块
  • 图片不能只保存文件路径，还要有尺寸、来源、所处位置、图像分析、OCR 结果

4. 诊断流

回答“这个对象哪里可能有问题，哪些地方不能太自信”。

  • diagnostics.json 或子目录诊断文件
  • 比如：这一页到底先读左边还是右边不太确定；这张图片里的字识别得不太准；这个表格边框信息不完整；这个图有图片文件但没找到图注。
  • 它不是正文，也不是结构本身，而是“风险提示”和“不确定性说明”。

说人话： 诊断流就像医生在检查报告旁边写的备注，不是在重复报告内容，而是在提醒你“这里可能看错”“这里证据不够”“这里需要人工再看一眼”。

如果没有诊断流，AI 很容易把不确定的信息当成确定事实。比如 OCR 其实只认对了七成，AI 却把它当成原文；或者页面是双栏排版，AI 把阅读顺序读反了。

为什么必须这样拆

AI 天生是偏纯文本的。即便多模态模型能看图，真正稳定、长上下文、可引用、可复核的，仍然要靠文本化、结构化后的证据包。

所以图片不能只是一个 img.png。表格不能只是“看起来像表格的一堆文字”。图注不能丢。段落层级不能丢。格式不能只剩一个“差不多”。

什么叫“像拆小汽车一样拆出来”

对象不能只拆成什么必须至少拆出什么
段落一整段纯文本段落块、runs、每段/每 run 的文字和格式、所属位置
标题“一行大字”标题级别、标题文本、块 ID、所属章节
表格按行拼起来的散文本表格块、行列、单元格内容、边框/宽度/合并信息、标题/图注关系
图片文件路径或二进制assetId、尺寸、来源页/位置、关联段落、图像结构分析、OCR 文字、图注关系
公式普通字符串公式块、来源、可见文本 fallback、是否 display、原始表示来源
批注/修订忽略不看批注内容、作者、锚点；修订类型、前后片段
PDF 页面整页 OCR 文本page、bbox、reading order、native text / OCR source、图块、表块、图注候选
PPT 一页整页文本拼接slide 结构、文本框、图片、表格、布局顺序、素材关联
Excel 表CSV 字符串sheet、range、表头、数据块、格式线索、后续图表/统计入口

什么叫“像组装小汽车一样装回去”

重装不是“把文本重新写一遍”。重装是：在保留用户原始意图的前提下，借助更规整的结构和格式，把文档重新落成一个更干净、更可维护、更可交付的对象。

场景允许的变化不允许的变化
老旧 Word 清理样式清掉脏样式、重复样式、错序 OOXML，重新规范字体/段距原来的表格消失、图注丢失、图片脱离上下文、段落顺序变掉
申报表整理表格边框更规矩、列宽更合理、字体统一、跨页更可读表格被打散成段落文本、表单字段失位
论文格式修复标题、正文、三线表、参考文献、行距更规范原有章节结构、公式、图表关系被破坏
PDF 转可编辑中间层把扫描件变成 page-aware、block-aware 的可读包只剩一整篇平铺文本，看不出页、图、表、图注、来源
PPT 读取后再生成按 slide 结构和元素重建只保留提纲文字，看不出每页布局和素材

最终的损耗标准

级别可以接受不能接受
内容损耗兼容预览里少量标点/空白差异；OCR 低置信字符保留 warning整段内容、整张表、图注、脚注、图片说明、页内顺序丢失
结构损耗极少数二级细节暂未精确映射，可明确标 warning标题树错乱、表格拆散、图片与图注脱钩、块 ID 不稳定
格式损耗在明确说明下，极少量高级继承样式或 Office 私有细节未完全恢复表格边框/续表/页边距/段距/字体混乱到改变用户原始意图
素材损耗图片结构分析不确定时给低置信或 null图片只剩文件，没有语义，没有位置，没有和正文的关系

育种类比：Nong 的交付标准就是“性状一致”

如果借用育种的话说，Nong 的目标不是把材料“处理过”就算完，而是要让重装后的对象保持性状一致。不能原本是一个规整的申报表、带层级的论文、带图注的报告，经过拆解和重装之后发生“性状分离”，最后变成只剩纯文本、表格散掉、图和图注脱钩、格式全乱的新对象。

性状层必须保持的一致性典型“性状分离”失败
语义性状原文要表达的内容、字段、图注、注释、公式、说明关系保持一致。文字还在，但图注没了；表格说明没了；公式退化成乱码字符串。
结构性状章节层级、表格结构、页面顺序、slide 顺序、sheet 结构保持一致。表格被拆成散段落；标题树乱了；页内阅读顺序颠倒。
格式性状用户原本依赖的可见格式意图被保住，并在必要时更规范。申报表原本靠表格表达，结果被优化成平铺文本；续表关系消失；字体段距失控。
素材性状图片、媒体、裁剪区域、OCR 文字、来源位置要保持可追踪。图片只剩文件名，不知道它在文档哪儿，也不知道它表达什么。

Nong 允许“规范化”，但不允许“变种化”。 样式可以清理，排版可以更规矩，边框可以更统一，但这些动作都只能是在原始意图之上做整理，不能把对象整理成另一个物种。

统一 slice 包，才是 AI 真正要读的对象

AI 不应该默认读原始 .docx / .pdf / .pptx / .xlsx。AI 应该默认读它们的 slice 包。

这套思想不是只适用于 Word

对象一刀入口三流如何体现最终可重装目标
Word word dissect 内容流看段落/表格/图注/图片；结构流看章节/块顺序；格式流看字体/段距/表格边框/页设置；资产流看图片与分析。 清理样式、修表、补内容、公文/学术格式化、保护、合并、追加元素。
PDF pdf dissect 内容流看 block 文本；结构流看页和阅读顺序；格式流看 bbox、字体、排版线索；资产流看页图、内嵌图、OCR 证据。 先对齐到可读可引用中间层，再与 Word 路线桥接，最终进入可编辑交付物。
PPTX pptx dissect 内容流看每页文字和表格；结构流看 slide、shape、顺序；格式流看布局和字体；资产流看图片和媒体。 未来应能进入 pptx create/edit 的重装路线。
Excel excel dissect 内容流看 cells/table；结构流看 workbook/sheet/table；格式流看表头、范围、基本格式；资产流看嵌入对象和图表线索。 未来应能从结构化 sheet/table 模型重装成更规范 workbook。
图片 ocr analyze-image / ocr local / ocr cloud 内容流是 OCR 文本；结构流是 region/block；格式流是位置/尺寸/方向；资产流是原图与裁剪图。 服务 Word/PDF/PPT 的复原与解释，不是孤立图片识别而已。

最终命令起名原则，也要服从这个核心

命令名不是随便取好听的，而是要和“拆、读、改、装”的产品动作一一对上。

词最终含义为什么适合 Nong
dissect把复杂对象切成统一包这就是“一刀”
slice读取切出来的包，而不是读原文件AI 的主入口应该是 slice，不是原始 Office/PDF
read轻量读取原对象某一部分适合快读，但不能代替 dissect
diagnose读包后给出综合判断比 check 更深，比 edit 更轻
validate判定对象是否过线对应质量闸门
write从规格写出一个内容对象适合论文、公文、信函这类语义对象
format把已有对象排成某种样子适合公文、学术格式、表格重排
audit只读审计证据，不改对象适合格式和质量验证
add在现有对象里追加一个明确类型的块适合 block-aware 重装
merge把多个对象安全拼装成一个对象适合重装路线，不只是文本拼接

AI 最应该怎么理解 Nong

用户说的话AI 不该直接想什么AI 应该先想什么
“帮我整理这个申报表”直接读纯文本然后重写先切片，保住表格、字段、图注、页面和格式证据，再决定怎么规范化
“帮我看看这个 PDF 讲了什么”直接 OCR 一整页文本先分类 text/hybrid/scan，再把页、图、表、阅读顺序、图片语义分开读
“把这个 DOCX 优化一下”用文本逻辑重写整篇先判断这是样式修复、结构修复还是内容追加；重装时不能破坏原布局意图
“给我做成规范的公文”只关心文字内容内容生成和版式整理是两步：inspect write-official 是写内容，word format-gongwen 是排版
“帮我看这张图是什么”只给一句图片描述如果图片属于文档对象，还要把它和 block、caption、page、bbox 绑定起来

最终设计下，各命令组该怎么站位

命令组最终角色在总蓝图里的位置
word复杂 Word 对象的拆、证据化、修、装一刀三流最成熟的主战场
pdf复杂 PDF 对象的拆、阅读顺序、页级证据化把 PDF 从黑盒变成可桥接对象
pptx复杂幻灯片对象的拆当前偏读，未来应进入可重装
excel复杂表格对象的拆和简化生成当前偏数据入口，未来应进入可重装 workbook
slice统一读包入口AI 最重要的阅读面
inspect读内容语义，生成语义对象建立在切片与文档对象之上，不代替切片
ocr补内容流和资产流中的图像文字/区域语义服务文档对象，不是孤立工具
chart / diagram把结构化数据或关系对象装成图偏“组装输出”，不是复杂输入拆解主线
skill保证 Toolkit 这层本身也是规范对象维护 AI 教学层的确定性

现在这套命令名，哪些是对路的，哪些要补更准的词

现名判断原因最终建议
word dissect对完全符合“一刀”的语义。保留
pdf dissect对也完全符合“一刀”的语义。保留
slice inspect对读的是包，不是原对象，很准。保留
word preview不够准听起来像视觉预览，实际上更像结构诊断。补 alias：word diagnose
word rebuild不够准真实动作是清理样式污染并规范化。补 alias：word clean-styles
inspect refs不够准太短，AI 容易误会成外部文献检索。补 alias：inspect references
inspect varplan不够准缩写太内部。补 alias：inspect variables
inspect data-req不够准缩写不适合教学。补 alias：inspect data-requirements
word format-gongwen对它是“已有对象的公文排版”，词和动作对得上。保留
inspect write-official对它是“从规格写公文草稿”，和排版职责区分清楚。保留

最终蓝图下，重复、缺少、不对该怎么判断

类别最终标准例子
重复同一个动作被两套主入口教给 AI，这才叫坏重复。word add paragraph 与 word add-paragraph 这种兼容 alias 不算坏重复。
缺少不是文案没写，而是“拆”和“装”的链条缺了一段。没有 pptx create，就说明 PPT 还没进入真正可重装阶段。
不对名字或 description 让 AI 猜错动作。word preview、inspect refs 这类。

最后的定稿口径

最后应该定成这样：

  • Nong 的核心不是“命令多”，而是“复杂对象一刀三流，拆得开，装得回去”。
  • AI 的主阅读对象不是原始 Office/PDF 文件，而是统一 slice 包。
  • content.nongmark 和 content.jsonl 是主要语义流；structure.json 是导航流；format.json 和 assets/manifest.json 是视觉/素材证据流。
  • 图片必须语义化，表格必须结构化，图注必须保留，格式必须留证据，回写必须控制损耗。
  • 命令名必须服从这条总产品逻辑：dissect 是切，slice 是读包，write 是按规格写内容，format 是排版已有对象，audit 是只读审计。

按这份蓝图，最先该定死的几件事

优先项为什么
把“一刀三流”写进所有对象的总原则，而不只是 Word 特性这决定整个产品是不是一个统一系统。
把 slice 包定成 AI 的默认阅读对象这决定 Toolkit 和 NanoBot 以后是不是稳定。
把“可重装 + 损耗控制”写成验收标准这决定 Nong 是不是生产工具，而不是只读解析器。
把不够准的命令补 alias，而不是继续靠口头解释这决定 AI 是否能从命令名直接猜对动作。
把未来缺口定义成“哪类对象还不能完整拆装”比按命令数凑功能更接近产品本质。
