1. 整体架构

用户输入 DSL
SU=('沸石'+'斜发沸石')*('缓释'+'控释')*('肥料'+'化肥')

       ↓ lit parse     词法分析 → 语法树
       ↓ lit plan      为每个 Provider 生成粗糙查询
       ↓ lit search    并行调用 Provider → 本地过滤 → 去重 → 排序
       ↓ lit export    导出 JSON / Markdown / BibTeX
       ↓ lit batch     扫描目录 → 批量执行 → 生成报告

Step 0: DSL 解析 (CNKI Parser)

将类 CNKI 检索表达式解析为抽象语法树 (AST)。支持 AND / OR / NOT / BETWEEN / 字段限定 / 括号分组 / 短语引用。

"SU=('沸石'+'斜发沸石')*('缓释'+'控释')*('肥料')"

  AndNode
  ├── OrNode (SU="沸石" | SU="斜发沸石")
  ├── OrNode (SU="缓释" | SU="控释")
  └── TermNode (SU="肥料")

+ = OR (或者) * = AND (并且) () = 分组

操作符含义示例
+OR (逻辑或)SU='A'+'B' → A 或 B
*AND (逻辑且)SU='A'*'B' → A 且 B
-AND NOTSU='A'-'B' → A 且不含 B
()分组SU=('A'+'B')*'C' → (A或B)且C
'...'短语SU='machine learning' → 精确短语
YE BETWEEN(2020,2024)年份范围2020-2024年间发表

Step 1: 检索计划 (Query Planner)

将 AST 转为 DNF (析取范式)，保留 AND 连接的概念组。去掉字段前缀（SU/TI/AB），将概念值按空格拼接为粗糙查询串。

AST:
  AndNode(OrNode("沸石","斜发沸石"), OrNode("缓释","控释"), TermNode("肥料"))
      ↓ DNF 展开
  DNF: [ ["沸石","缓释","肥料"], ["沸石","控释","肥料"],
         ["斜发沸石","缓释","肥料"], ["斜发沸石","控释","肥料"] ]
      ↓ 去字段前缀，空格拼接
  Rough Queries:
    "沸石 缓释 肥料"          ← 发往 OpenAlex/Crossref/AMiner
    "沸石 控释 肥料"
    "斜发沸石 缓释 肥料"
    "斜发沸石 控释 肥料"

DNF 展开保证：如果 DSL 里写了多个 OR 组，管线会把所有组合都搜一遍，不丢信息。

Step 2: 候选 (Provider Search) — 粗搜

并行调用各 Provider 的 REST API，拿回粗搜结果。每个 Provider 独立调用、独立返回，不互相依赖。

ProviderAPI语言认证速率
OpenAlex api.openalex.org/works EN 为主 NONG_LIT_OPENALEX_API_KEY (推荐) 50 req/s (有 key)
Crossref api.crossref.org/works EN 无需认证 (NONG_LIT_MAILTO 可选) 10 req/s (polite pool)
AMiner datacenter.aminer.cn 中英文 NONG_LIT_AMINER_KEY (JWT) 免费层级
Metaso (秘塔) metaso.cn/api/v1/search 中英文 NONG_LIT_METASO_KEY (Bearer, mk-) 新用户 5000次
Unpaywall api.unpaywall.org/v2 EN NONG_LIT_MAILTO (必填) DOI 查询
注意: Provider 粗搜只看关键词匹配，不会做 DSL 级别的精确过滤。比如搜 "沸石 缓释 肥料"，Google/Crossref 会返回任何包含这些词的论文——哪怕那篇论文跟"沸石分子筛"没有任何关系。精确过滤在 Step 4 本地做。
关键设计: 所有 Provider 都是纯 C# HttpClient 直接调 REST API。没有 Python / MCP / SSE 中间层。AMiner 在 v4.2.2 中从 Python MCP 改为纯 C# REST Provider。

Step 3: 合并去重 (Paper Merger)

三源回来的论文可能有重叠——同一篇论文 OpenAlex 有、Crossref 也有。Merger 负责去重。

策略优先级
DOI 匹配最高 —— DOI 相同 → 同一篇
标题+年份+第一作者备用 —— 无 DOI 时用。中文论文额外使用字符相似度

去重后，字段合并：OpenAlex 的摘要 + Crossref 的引用数 + AMiner 的中文标题——信息最丰富的那份被保留。

Step 4: 本地过滤 (Local Boolean Filter)

这是管线最核心的区别点。Provider 粗搜返回的结果里有很多噪音。本地过滤器把原始 DSL 的字段约束逐条应用到每条论文。

DSL: SU=('沸石'+'缓释')*('肥料')

每条论文检查:
  "沸石" 或 "缓释" 是否在 Title/Abstract/Keywords/Concepts 中?  → YES/NO
  AND
  "肥料" 是否在 Title/Abstract/Keywords/Concepts 中?             → YES/NO
  → 两者都 YES → 保留。任一 NO → 丢弃。
字段搜索范围
SU (Subject, 默认)Title + Abstract + Keywords + Concepts + Topics
TITitle only
ABAbstract only
AUAuthor names
KWKeywords
JNJournal/Venue name
YEPublication year
CFCitation count
DOIDOI exact match
FIFirst author
AFAffiliations

两种过滤模式

模式行为适用场景
strict 严格校验。字段缺失或中文词无法匹配英文摘要 → 丢弃 英文文献精确检索
recall 宽松校验。CJK 检索词在英文摘要中找不到时保留论文(标记 recall-cjk-mismatch)。FT 字段缺失时也保留。 中文 DSL × 英文文献 (默认)

lit batch 自动使用 recall 模式。

Step 5: 排序 (Literature Ranker)

多因子加权打分，得分靠前的排在前面。

因子权重说明
概念覆盖度40%DSL 查询词在论文中命中的比例（越高越好）
引用数归一化25%论文被引次数 / 年份归一化（越老允许越少引用）
时效性20%越新越高，25 年跨度衰减
来源质量15%AMiner > OpenAlex > Crossref（中文覆盖 + 数据质量）

排名配置文件（--profile）：

Profile倾向
balanced (默认)综合平衡
classic偏向高引用经典文献
recent偏向最新文献

排序后截取 Top N（--limit 控制），返回给用户。

3. 全部 CLI 命令

命令用途示例
nong lit parse 解析 DSL 语法，验证合法性 nong lit parse --query "SU='艾草'*'种植'" --json
nong lit plan 展示每个 Provider 生成的粗糙查询计划 nong lit plan --query "SU=..." --json

4. lit batch 工作流

目录结构:
搜索文献的东西/
  ├── 矿物基复合材料...txt   ← 含 2 条 DSL
  ├── 土传病害...txt         ← 含 2 条 DSL
  ├── 环境安全性...txt       ← 含 1 条 DSL
  ├── 采前处理...txt         ← 含 1 条 DSL
  └── 草案.txt               ← 跳过 (无 DSL)

一条命令:
nong lit batch 搜索文献的东西/ -o report.md --limit 5

输出:
report.md
  ├── ## 矿物基复合材料
  │   ├── ### 沸石缓释体系
  │   │   • DSL: SU=('沸石'...)*('缓释'...)*('肥料'...)
  │   │   • 候选 99 → 去重 6 → 返回 5
  │   │   • 论文1: Zeolite slow-release...
  │   │   • 论文2: ...
  │   └── ### 腐植酸-稀土体系
  │       ...
  ├── ## 土传病害
  │   ├── ### 病原菌抑制
  │   └── ### 根际微生态
  ├── ## 环境安全性
  ├── ## 采前处理
  └── (自动汇总)
批量检索的完整流程: 写 DSL 在 .txt 里 → 跑 nong lit batch → 打开 report.md → 筛选你想要的文献 → 粘回 txt 整理引用列表。从数天的手工检索变成一条命令。

7. 设计原则

  1. Provider 粗搜 + 本地精滤: Provider 的 API 搜素能力有限（多数只支持简单关键词），不做 DSL 级别的过滤。管线先让 Provider 宽搜，再用本地过滤器按 DSL 语义精确筛选。
  2. 纯 C#，零外部依赖: 所有 Provider 都是 HttpClient 直接调 REST API。没有任何 Python 脚本、MCP 协议、Node.js 进程。
  3. DNF 展开保证完整性: OR 组在发往 Provider 之前展开为所有组合，确保搜索不丢信息。
  4. recall 模式处理中英混杂: 中文 DSL 检索英文文献时，provider 粗搜返回的论文里未必有中文关键词，本地精确匹配会失败。recall 模式在 CJK 查询词与纯英文摘要不匹配时保留论文。
  5. 去重优先 DOI，降级标题+作者: DOI 是唯一标识符，无 DOI 时用标题+年份+第一作者去重。
